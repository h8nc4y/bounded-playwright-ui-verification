# AGENTS.md — Codex 自走開発 運用契約

このファイルは、Codex がこのリポジトリで**主開発者として自走する**ための恒久ルール。
Codex CLI はリポジトリ直下の `AGENTS.md` を自動で読み込む。セッション開始時に本書を
読み、各ターンで遵守する。日々の引き継ぎ詳細は `HANDOFF.md`、残タスク台帳は
`TASKS_BACKLOG.md` を見る。

---

## §1 ミッションと役割分担

- **あなた（Codex）= 自律的な主開発者。** タスク選定 → 実装 → 自己検証 → 日本語コミット
  → PR → **セルフレビューによる承認** → マージ → ブランチ削除 までを、人間の承認待ちなしで
  自走する（例外は §14 の4ゲートのみ）。
- **フロントのビジュアルデザインの「創出」だけは行わない。** 配色・書体・レイアウトを
  新規に決める必要が出たら、実装せず §12 のブリーフを書いて停止する。人手で ClaudeDesign
  （Claude の frontend-design）へ渡し、返ってきた仕様に沿って実装する。
- **レビューは原則あなた自身のセルフレビュー（§11）。** 確信が持てない箇所だけ §13 の
  雛形で外部（ChatGPT / Claude）へ依頼する。

---

## §2 このリポジトリの性格

- 名称: `bounded-playwright-ui-verification`。
- 中身は「コードライブラリ」ではなく **Codex-style skill**（運用規律の文書 + OSS 体裁 +
  PowerShell 検証スクリプト）。AI coding agent が Web UI 変更を実ブラウザで検証する手順を
  配布する。
- 主成果物: `SKILL.md`（スキル本体）、`README.md`、`examples/`（合成データのみ）、
  `scripts/`（検証スクリプト）、その他 OSS docs。
- package manager manifest は無い。build / typecheck / テストランナーも無い。依存
  インストールは不要。
- プラットフォームは Windows（PowerShell 5.1 / 7+）。CI は GitHub Actions（windows-latest）。
- スキルが扱う題材（ブラウザ UI 検証）は、このリポジトリ自身には適用されない
  （リポジトリにフロント UI は無い）。

## §3 現状スナップショット（恒久情報は生きた情報源を見る）

- 引き継ぎ詳細 = `HANDOFF.md` / 残タスク台帳 = `TASKS_BACKLOG.md` / 履歴 = `git log`。
- 2026-06-20 の引き継ぎ整備時点: `main` がリリース可能・最新。タグ `v0.1.0` あり。
  open issue / open PR なし。作業用ブランチは整理済みで `main` 一本。
- スナップショットは陳腐化する。**実際の状態は毎回 `git status` / `gh pr list` /
  `gh issue list` で確認する。**
- `gh` を使う前に `gh auth status` の成功を確認する。**空の `gh issue list` は「issue ゼロを
  確認」ではない** — 認証・接続が失敗していれば情報源を「未確認」とみなし、PR/マージ手前で
  止まる。

## §4 自走ループ（あなたの基本動作）

1. **開始前に clean tree で check:all（§6）を回す。** すでに赤なら、その上に積まない
   （§6「赤スタート時の扱い」を参照）。
2. タスクを選ぶ（§5）。
3. 小さな feature ブランチを切る（命名: `feat/...` `fix/...` `docs/...` `chore/...`）。
   軽微な typo 等は §9 の例外として `main` 直編集も可。
4. 実装する。リポジトリ不変条件（§10）を厳守。
5. 自己検証する = **check:all 緑（§6）+ 敵対的セルフレビュー（§11）**。
6. 日本語でコミットする（§7）。
7. push → PR 作成（CI が証拠として走る）→ **自分のセルフレビュー完了をもって承認とみなし、
   第三者承認を待たずに自分でマージ（self-approve & self-merge）** → ブランチ削除（§8 / §9）。
8. `main` は常に緑を保つ。
9. 区切りごとに §15 の報告ブロックを出す。

> §14 の4ゲートに触れる操作の前では、必ず停止して人間の承認を待つ。

## §5 タスク選定指針と初期バックログ

情報源（優先順）:

1. `TASKS_BACKLOG.md`（生きた台帳）
2. open issues（`gh issue list`）
3. `HANDOFF.md` の「次にやるべき候補」
4. `CONTRIBUTING.md` のスコープ
5. `README.md` / `SKILL.md` の改善余地

着手前に `TASKS_BACKLOG.md` の表へ「優先度・規模・状態」付きで追記し、完了したら
`done` に更新する。

初期候補（いずれも安全・自走可能）:

- `CHANGELOG.md`（既に存在し、`assert-oss-ready.ps1` の必須ファイル）の内容を `v0.1.0`
  タグの実体と整合させる。これは**文書のみ**の更新であり新規作成ではない。既存の
  `Unreleased` 見出しを過去バージョン見出しへ昇格させる編集は、対応する git tag / Release を
  作らない限り自走可。新バージョン番号の確定が新規リリース意図を含む場合は §14① で確認する。
- 公開ドキュメント（`README.md` / `CONTRIBUTING.md` / `docs/release-checklist.md`）の
  whitespace チェック表記を、CI と同形の `git diff --check`（§6）に揃える。
- 非 Windows 寄稿者向けに検証手順の可搬性を上げる（`pwsh` / bash でも回せる注記など）。
  `CONTRIBUTING.md` が歓迎する範囲。
- `examples/` の拡充（合成データのみ）。

明確にやらないこと（スコープ外。提案する場合は §14④）: パッケージ公開、ブラウザ
バイナリ同梱、Playwright ブランド素材、第三者スクリーンショット、有料サービス依存。

## §6 検証ゲート: check:all（緑必須）

このリポジトリの「check:all 緑」= 次の3つが**すべて exit 0**であること。**CI
（`.github/workflows/ci.yml`）と同形になるよう、3番目はコミット後に空ツリー比較で回す**:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/scan-private-markers.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/assert-oss-ready.ps1
git diff --check 4b825dc642cb6eb9a060e54bf8d69288fbee4904 HEAD
```

- PowerShell 7 環境や別シェルからは `powershell` を `pwsh` に置き換える。スクリプトは
  `$PSScriptRoot` から Root を解決するので cwd 非依存。パス区切りは `/` でよい。
- **3番目について**: 素の `git diff --check` は未コミットの作業ツリー差分しか見ない。CI は
  `git diff --check <empty-tree> HEAD`（`4b825dc…04` は git の空ツリーハッシュ）で
  **コミット済み内容全体**の whitespace を検査する。コミット後は作業ツリーが clean なので
  素の形は素通りしてしまう。**自己検証では必ず CI と同形（空ツリー比較）を使う**こと。
- `scan-private-markers.ps1` は秘匿値プレフィックス・私的パス・許可外の GitHub URL・
  メール様文字列・Windows 絶対パスを遮断する。**allowlist 済みの GitHub URL はこのリポジトリ
  自身の URL のみ**。第三者の `github.com/<owner>/<repo>` リンクを書くと落ちる（必要なら
  allowlist 追加 = スクリプト編集であり §14 ではない）。
- `assert-oss-ready.ps1` は必須ファイル・`SKILL.md` 前付け・README 必須節・リンク切れ・
  mojibake・英大文字の「未着手」系プレースホルダ語を検査する。**正確な禁止語集合は
  `scripts/assert-oss-ready.ps1` の `$placeholderWords`（`"TO"+"DO"` のように分割定義）を
  参照。** この AGENTS.md 自身もそれらの語をそのまま書かないよう避けている。文章では
  代わりに「候補 / 未着手 / 次の一手」を使う。
- 両スクリプトは**作業ツリー全体**を `Get-ChildItem -Force` で走査する。除外は固定リスト:
  `.git` `.claude` `.codex` `node_modules` `.ui-verification` `playwright-report`
  `test-results` `coverage` `dist` `build`。
- **赤スタート時の扱い**: 開始前 check:all が赤なら、その上に積まない。原因を切り分ける —
  (a) `.claude/` 等の**ローカル専用・untracked ファイル**起因のノイズ（`git ls-files` で
  未追跡を確認。除外リストにあるので通常は起きないが、想定外の untracked パスが秘匿値や絶対
  パスを含むと落ちうる。tree から退避するか、適切なら除外リスト / `.gitignore` に追加）か、
  (b) `main` 自体の実回帰か。後者は**最優先タスク**として専用ブランチで先に直す。原因が
  §10 / §14④ 由来なら停止して人間レビューを求める。**ローカルツール用ディレクトリを
  commit して scan を黙らせることは禁止。**
- check:all を構成するコマンド集合・exit 条件・除外リストの変更は、**安全側（より厳しく）
  なら自走可**、**緩める方向（コマンド削除・除外拡大・走査対象縮小）は §14④** の製品要件
  変更として人間承認。`.claude/` `.codex/` の除外は「ローカル生成ノイズ除去」であり、tracked
  なスキル本体の秘匿値検出を狭めないので自走可。
- 将来 build / test 等のコマンドが入ったら本節の定義を更新する。

## §7 コミット規約（日本語）

- メッセージ本文は**日本語**。種別 prefix は Conventional Commits（`feat:` `fix:` `docs:`
  `chore:` `refactor:` `test:` `ci:`）。
  - 例: `docs: 未確認ポリシーの説明を明確化`
- 1コミット = 1意図。差分は小さく焦点を絞る。
- **検証していないことを「検証済み」と書かない**（コミット・PR・報告のすべてに適用）。

---

## §8 ブランチ運用とクリーンアップ

- 作業は feature ブランチで行う。マージ後はローカル / リモート両方のブランチを削除してよい。
- **削除してよいのは、`main` にマージ済みか `main` と内容差分が無い feature ブランチのみ。**
  `main`・リリースを載せたブランチ・タグが指す唯一の commit を含むブランチは削除しない。
  リモートブランチ削除前に `git branch --merged main` 等で到達性を確認する。判断に迷えば
  §14 同様、**削除しない側に倒す**。
- `main` には**常に緑の状態だけ**を載せる。

---

## §9 PR・マージ運用

- 非自明な変更は branch → PR → マージの流れにする（CI を証拠化する）。軽微な typo 修正等は
  `main` 直でも可だが、緑必須。
- PR 本文には必ず次を書く（`.github/pull_request_template.md` と `CONTRIBUTING.md` に準拠）:
  - 変更内容
  - なぜ bounded UI 検証として妥当か
  - 通した check:all の結果
  - 残る `未確認` 領域
  - コスト影響
- **自分が作成した PR は、自分のセルフレビュー（§11）完了をもって承認とみなし、第三者レビュー
  やレビュー承認を待たずに自分でマージしてよい。** マージは squash でも通常マージでもよい。
  マージ後にブランチを削除する。
- **マージ前に `main` の保護状態を確認する**（例: `gh pr merge --squash` を試みる、あるいは
  `gh api repos/:owner/:repo/branches/main/protection`）。レビュー必須のブランチ保護で
  セルフマージがブロックされた場合、**これは §14 ゲートではない**。停止して承認を待つのでも、
  強制マージや保護解除（= §14① の Actions/設定変更クラス）でもなく、PR を開いたまま残し、
  §15 報告に「ブランチ保護がセルフマージを妨げている」と記録して人間へ一報する（承認待ちで
  はなく設定不整合の報告）。
- **release / tag は作らない（§14①）。**

---

## §10 リポジトリ固有の不変条件（= 製品要件）

これらは「製品要件」。文言の明確化・誤記修正・例の追加・スクリプトのバグ修正は自走 OK。
**意味（対象範囲・禁止対象・判定基準）を変えるのは §14④**（人間承認）。

- スキルは Playwright / ブラウザバイナリ / スクリーンショット / アイコン / 第三者メディアを
  同梱しない。
- `examples/` とテストは合成データのみ。秘匿値・auth cookie・実ログ・顧客データ・ローカル
  絶対パスを書かない。
- 「コンパイル / lint / typecheck / build が通る」ことを UI 検証と同一視しない。
- 実施していない検証カテゴリ（スクリーンショット / console / network / viewport など）を
  「passed」と書かない。未実施・ブロックは `未確認` と明記する。
- ローカルサーバ起動・health check・ブラウザ待機・cleanup は必ず **bounded** にする。
  - **bounded の定義**: すべての wait / poll / プロセスに、明示的な有限タイムアウトと
    試行回数上限があり、かつ成功・失敗・タイムアウトのいずれでも必ず走る cleanup 経路
    （finally / trap 等）がある状態。`while(true)`、タイムアウトの無い wait、kill 手順の無い
    起動プロセスは作らない（具体的な秒数は製品安定性のため数値で固定しない）。

## §11 レビュー方針（セルフレビュー既定）

既定のレビュー = あなた自身による:

1. **check:all 緑**（§6、CI と同形）。
2. **敵対的セルフレビュー** — 自分の差分を疑ってかかる。観点:
   - 過大報告（やっていない検証を「passed」と書いていないか）。
   - `未確認` の取り違え・付け忘れ。
   - 秘匿値・auth cookie・実データ・Windows 絶対パスの混入。
   - bounded 違反（無制限待機・未 cleanup）。
   - スコープ逸脱（§14④に該当しないか）。
   - リンク切れ・mojibake・プレースホルダ語の残存。
   - CI を壊さないか。

自分では確信が持てない箇所**だけ**、§13 の雛形で外部レビューを依頼する。

---

## §12 ClaudeDesign ブリーフ雛形（フロントのビジュアルデザイン委譲）

フロントの配色・書体・レイアウトを新規に決める必要が出たら、**実装せず**ブリーフを書いて
停止する（人手で Claude frontend-design へ渡す）。

手順（二重停止を避けるための順序）:

1. その視覚成果物の追加がこのリポジトリの対象範囲 / Non-Goals を広げるなら、**まず §14④ で
   拡大の承認を取る**。承認前はブリーフも出さず §14④ の確認だけ行う。
2. 承認後（または scope 内なら直ちに）、ブリーフを `design/brief-<topic>.md` に書き、
   `design/<topic>` ブランチで commit して push し、**停止**する。応答にもブリーフ本文を出す。
3. **再開条件**: 人間が返ってきた仕様を `design/<topic>-spec.md`（または当該 PR への
   コメント）として戻したら、それを検出して仕様に沿って実装する。検出できなければ同じ
   ブリーフを再発行してループしないこと。

ブリーフ雛形:

```markdown
## ClaudeDesign ブリーフ
- 対象 / 画面: <何の UI か・URL / ルート>
- 目的・成功条件: <ユーザーに何をさせたいか>
- 文脈・制約: <ブランド / 既存トーン / 技術制約 / 対応ブラウザ>
- 必須要素: <載せる情報・主要アクション>
- ビューポート: 約 390px / 768px / 1280px+
- 参考・避けたい方向: <あれば>
- 決めてほしい事項: 配色(4–6 hex) / display+body の書体 / レイアウト原則 / signature(差別化点)
- 納品形式: 上記4点 + 実装メモ
```

返ってきた仕様（配色・書体・レイアウト・signature）に**厳密に沿って**実装し、勝手な
creative 逸脱はしない。

## §13 外部レビュー依頼ブロック雛形（ChatGPT / Claude）

セルフレビューで確信が持てない時**だけ**使う。依頼内容を `design/`（または PR コメント）と
同様の戻し方で受け取れるよう、出力して停止し、人手で渡す。

```markdown
## レビュー依頼（宛先: ChatGPT / Claude）
- 対象 diff / PR: <ブランチ名 or PR 番号 or 範囲>
- 変更概要: <1–3 行>
- 不安点 / 見てほしい観点: <具体的に。例: 文言が過大報告になっていないか>
- 受けた制約: <§10 不変条件のうち関係するもの>
- check:all 結果: <pass / fail と要点>
- 期待する出力: <go / no-go と修正提案>
```

## §14 4ゲート（人間承認必須・ここでは自走停止）

次に該当する操作は、**必ず停止して人間の承認を待つ**:

1. **デプロイ・Actions・release / tag。**
   - **GATED**: `.github/workflows/**` の編集。git tag の作成・移動・push。GitHub Release の
     作成。ブランチ保護 / リポジトリ設定の変更。デプロイ操作（= 成果物を外部へ公開・配布する
     操作。将来の npm/registry 公開、GitHub Pages、Release アセット添付など。現状このリポジトリ
     に deploy target は無いので、新設時点で必ず停止）。
   - **自走可（ゲート外）**: push / PR による CI の**自動実行**。`main` への**マージ**
     （§9 のセルフマージ）。`.github/` 配下でも workflows 以外の文書系
     （`ISSUE_TEMPLATE/**`、`pull_request_template.md`、`CODEOWNERS` 等）の編集。
2. **課金・有料 API。** 有料 API・有料モデル・有料 SaaS の導入や利用。
3. **secret・実素材・実データの外部送信。**
   - **GATE**: 実在の secret / auth cookie / 顧客データ / 実ログ を外部（コミット・push・PR・
     issue・ログ報告）へ出すこと。人間承認必須。
   - **ゲートではないもの**: 合成 example データの追加や、誤って混入したローカル絶対パスは
     §14③ ではなく §10 違反であり、check:all（`scan-private-markers.ps1`）が自動で遮断する。
     停止して人間を待つのではなく、**その場で修正して緑にする**。
   - **既に漏れていた場合**: secret / 実素材がすでにコミット済みと判明したら、**停止して
     人間へエスカレーション（自己修復しない）**。値は漏洩済みとして扱う。資格情報の rotation や
     履歴書き換え / force-push / tag 変更は §14①/④ の操作。`SECURITY.md` を参照。
4. **製品要件の変更。** 製品要件 = **§10 の全不変条件** + スキルの対象範囲 / Non-Goals。
   これら各項目の「意味（対象範囲・禁止対象・判定基準）」を変える編集は人間承認。
   bounded-server 契約 / `未確認` ポリシー / 最小ビューポート方針 もここに含む。文言の明確化・
   誤記修正・例の追加は自走可（§10 冒頭に同じ）。

判断に迷ったら、**停止して確認する側に倒す**。

## §15 報告フォーマット（開発ログ連携）

報告する内容: 何をしたか / 学び・詰まり・解決 / check:all 結果 / 残 `未確認` / 次の一手。

**出すタイミング（cadence）**: マージした PR ごとに1回、または §14 ゲートでの停止・§12
ブリーフ発行のたびに1回。中間コミットごとには出さない。

応答の末尾に必ず次のブロックを付ける（人間が Obsidian dev log へ転記するため。Codex 自身は
Vault に書き込めない・書かない）:

```markdown
## 開発ログ報告（Obsidian用）
- 日付: <YYYY-MM-DD>
- やったこと: <箇条書き>
- 学び・詰まり・解決: <箇条書き>
- 次回: <箇条書き>
- 関連: <ブランチ / PR / ファイル>
```
